# GNUstep Audit Instrumentation Suite

Regression tests and performance benchmarks for the GNUstep audit findings.
51 regression tests across 7 repos validate that fixes are correct.
14 performance benchmarks measure optimization impact quantitatively.

## Quick Start

```bash
# Run all regression tests
make tests

# Run all performance benchmarks
make benchmarks

# Save benchmark results as baseline (before applying fixes)
make baseline

# Run benchmarks and compare against saved baseline (after applying fixes)
make compare

# Run everything
make all

# See all available targets
make help
```

All commands run in an MSYS2 bash environment on Windows.

## Workflow

The intended workflow for validating fixes:

1. Build and install the unmodified GNUstep libraries
2. `make baseline` -- saves benchmark numbers as the reference point
3. Apply the audit fixes to the GNUstep source
4. Rebuild and install the modified libraries
5. `make compare` -- runs benchmarks again and shows delta vs baseline
6. `make tests` -- confirms all regression tests pass with the fixes applied

The compare target flags any benchmark that regresses more than 5% as SLOWER.

## Test Naming Convention

Test files follow the pattern `test_<finding_area>.m`, where each file targets
a specific finding ID from the audit:

| Pattern | Example | Maps to |
|---------|---------|---------|
| `test_<area>.m` | `test_rethrow_uaf.m` | RB-1: exception rethrow use-after-free |
| `test_<area>.m` | `test_secure_coding.m` | RB-1: NSSecureCoding enforcement |
| `test_<area>_race.m` | `test_selector_race.m` | TS-2/TS-3: selector table concurrency |

Each test file contains a comment at the top identifying the exact finding ID(s)
it validates. Tests use the conventions of their parent repo:

- **libobjc2**: `assert()` based, built with CMake
- **libs-base, libs-gui, libs-corebase**: `Testing.h` macros (`PASS`, `PASS_EQUAL`, etc.)
- **libs-opal, libs-quartzcore**: Standalone tool programs with assert/exit

## Benchmark Naming Convention

Benchmark files follow the pattern `bench_<area>.m`, where each file targets
a specific optimization area from the Phase 6 deep-dive:

| File | Measures |
|------|----------|
| `bench_msg_send.m` | Message dispatch throughput (objc_msgSend) |
| `bench_retain_release.m` | Retain/release cycle throughput |
| `bench_weak_ref.m` | Weak reference ops at 1/2/4/8 threads |
| `bench_autorelease.m` | Autorelease pool push/pop/drain |
| `bench_string_hash.m` | NSString hash for short/medium/long strings |
| `bench_dict_lookup.m` | NSDictionary lookup for small/large dicts |
| `bench_json_parse.m` | JSON parse throughput (MB/s) |
| `bench_runloop_timers.m` | Timer fire latency with N timers registered |
| `bench_nscache.m` | NSCache get/set throughput |
| `bench_cfarray_append.m` | CFArray sequential append (O(n) vs O(n^2)) |

Each benchmark uses the `BENCH` and `BENCH_JSON` macros from `common/bench_harness.h`
for consistent timing and output format.

## How to Add a New Test

1. Identify the finding ID (e.g., RB-5) and the repo it belongs to (e.g., libs-base)
2. Create `tests/<repo>/test_<descriptive_name>.m`
3. Add a comment header with the finding ID:
   ```objc
   /* test_archive_bounds.m -- RB-5: archive index bounds checking
    * Validates that NSUnarchiver rejects out-of-bounds cross-references.
    */
   ```
4. Write the test using the repo's conventions:
   - For libs-base/gui/corebase: `#include "Testing.h"` and use `START_SET`/`END_SET`/`PASS`
   - For libobjc2: use `assert()` and return 0 on success
   - For opal/quartzcore: standalone program, `exit(1)` on failure
5. Add the test program name to the repo's `tests/<repo>/GNUmakefile`
6. Run `make tests-<repo>` to verify it works

## How to Add a New Benchmark

1. Create `benchmarks/bench_<area>.m`
2. Include the shared harness:
   ```objc
   #include "../common/bench_harness.h"
   ```
3. Use the `BENCH` macro for human-readable output and `BENCH_JSON` for
   machine-readable output:
   ```objc
   int main(int argc, char *argv[]) {
       int json = (argc > 1 && strcmp(argv[1], "--json") == 0);

       // Setup code here...

       if (json) {
           BENCH_JSON("my_benchmark", 1000000, {
               // code to measure
           });
       } else {
           BENCH("my_benchmark", 1000000, {
               // code to measure
           });
       }
       return 0;
   }
   ```
4. Add the program name to `benchmarks/GNUmakefile`
5. Add the program name to the `BENCH_PROGRAMS` list in the master `Makefile`
6. Run `make benchmarks` to verify

## Results Format

Benchmark results are stored as JSON arrays in `benchmarks/results/`:

```json
[
{"name":"msg_send_cached","ops_per_sec":250000000.0,"ns_per_op":4.0,"iterations":10000000}
,
{"name":"retain_release","ops_per_sec":80000000.0,"ns_per_op":12.5,"iterations":5000000}
]
```

The `make compare` target reads `baseline.json` and `current.json`, then prints
a table showing the delta for each benchmark. Any regression over 5% is flagged.
